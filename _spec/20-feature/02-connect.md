# Screen 02 – 커플 연결
## Trust-Based Pairing Screen

---

## 1. 화면 목적 (Why)

이 화면의 목적은 “연결”이 아니라 **신뢰 형성**이다.

사용자가 느껴야 할 감정:
- “이 앱은 관계를 존중한다”
- “통제 구조를 강요하지 않는다”
- “상대에게 부담을 주지 않는다”

👉 기능 설명 ❌  
👉 관계 설계 ⭕

---

## 2. 행동 정책 (Behavior Policy)

### B1. 관계는 ‘초대’이지 ‘등록’이 아니다
- 커플 연결은 소유 관계가 아니다
- 상호 동의 기반으로만 성립한다

### B2. 역할은 통제 권력이 아니라 책임이다
- 관리자 / 실행자는 **상하 관계가 아님**
- 관리자는 “감시자”가 아니라 “맡아주는 사람”

### B3. 부담 최소화 원칙
- 이 화면에서는:
  - 목표 설정 ❌
  - 일정 입력 ❌
  - 습관 설명 ❌

> 연결 = 준비  
> 행동은 다음 화면에서 시작

---

## 3. 사용자 흐름 (User Flow)

### 3.1 진입 조건
- 로그인 완료
- `coupleId` 미존재 상태

---

### 3.2 연결 방식 (MVP)

#### 방식 A. 초대 코드 생성 (권장)
- 사용자 A:
  - “초대 코드 만들기”
  - 8자리 코드 생성 (숫자/영문 대문자)
- 사용자 B:
  - “초대 코드 입력”
  - 코드 입력 후 연결 요청

#### 연결 완료 조건
- 양쪽 모두 연결 승인
- `coupleId` 생성

---

## 4. 역할 확정 정책

### 4.1 역할 지정 시점
- 연결 직후 1회만 진행
- 이후 변경 불가 (MVP 기준)

### 4.2 역할 선택 방식

#### 옵션 1 (권장)
- 초대자 = 관리자
- 참여자 = 실행자

#### 옵션 2
- 역할 선택 화면 제공:
  - “이번엔 내가 맡을게 (관리자)”
  - “이번엔 내가 해볼게 (실행자)”

> 문구에서 ‘권한’ 단어 사용 금지  
> ‘책임’과 ‘맡김’ 표현 사용

---

## 5. UI 구성 요소

### 5.1 메인 메시지 영역

#### 헤드라인 (1줄)
- “이번 달, 누군가에게 맡겨볼래요?”

#### 서브 카피
- “계획은 내가 세우고,
  관리는 상대가 해주는 방식이에요.”

❌ “관리자 / 실행자 설명” 장문 금지

---

### 5.2 초대 코드 영역

#### 초대자 화면
- 코드 카드 UI
- 버튼:
  - “코드 복사”
  - “공유하기”

#### 참여자 화면
- 코드 입력 필드 (8칸)
- 버튼:
  - “연결 요청 보내기”

---

### 5.3 연결 대기 상태

- 메시지:
  - “상대가 연결을 확인 중이에요”
- 애니메이션:
  - 매우 미세한 로딩
- 취소 버튼 없음 (압박 방지)

---

- 코드 입력은 “정돈된 느낌”

---

## 7. 마이크로 카피 가이드

| 상황 | 금지 | 권장 |
|---|---|---|
| 연결 | 등록 | 연결 |
| 역할 | 권한 | 맡기기 |
| 관리자 | 감독 | 함께 관리 |
| 실패 | 잘못됨 | 다시 시도해볼까요 |

---

## 8. 예외 처리 정책

### 코드 오류
- 메시지:
  - “코드가 맞지 않아요. 다시 확인해볼까요?”
- 절대 사용 금지:
  - “유효하지 않은 코드”

### 연결 취소
- MVP에서는 제공하지 않음
- 관계 안정성 우선
