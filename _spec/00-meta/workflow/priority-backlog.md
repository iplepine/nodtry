# Priority Backlog

## 목적

현재 코드베이스와 테스트 상태를 기준으로, 다음 작업 우선순위를 실행 가능한 수준으로 정리한다.

## P0

### 1. History 필터 즉시 반영 안정화

- 문제: 히스토리 탭에서 필터 변경 시 상태는 바뀌지만 스트림 재계산이 보장되지 않아 UI가 즉시 갱신되지 않을 수 있다.
- 대상: `HistoryViewModel`, 히스토리 탭 필터 UI, 관련 테스트
- 작업:
  - 필터 변경 시 재계산 또는 재구독이 확실히 일어나도록 구조 수정
  - `all / me / partner` 케이스 위젯 테스트 추가
- 완료 조건:
  - 필터 탭 전환 시 리스트가 즉시 변경된다
  - 테스트로 필터별 결과가 보장된다

### 2. Now 탭 초기화 및 테스트 멈춤 현상 해결

- 문제: 앱 시작 시 `completeOverduePlans()` 예외 처리와 스트림 초기화 구조 때문에 테스트가 멈추는 구간이 있다.
- 대상: `NowTabViewModel`, 테스트 더블, 시작 시점 플로우
- 작업:
  - `completeOverduePlans()` 실패를 앱 흐름과 분리
  - 테스트용 Mock Repository에 필요한 구현 보강
  - `flutter test` 전체가 정상 종료되도록 스트림 초기화 안정화
- 완료 조건:
  - `flutter test`가 중단 없이 끝난다
  - 시작 시점 예외가 화면 로딩을 막지 않는다

## P1

### 3. 관리자 반려(Reject) 플로우 완성

- 문제: 관리자 승인 흐름은 있으나 반려 흐름은 MVP 기준으로 미완성이다.
- 대상: 계획 제안 카드, 상태 전이, 저장소 로직, 알림 정리
- 작업:
  - `reject` UX와 상태 전이 정의
  - 반려 사유 입력 또는 선택 흐름 추가
  - 반려 시 알림 및 후속 화면 상태 정리
- 완료 조건:
  - 관리자가 계획을 반려할 수 있다
  - 반려된 계획 상태가 일관되게 저장된다
  - 관련 테스트가 존재한다

### 4. 승인된 계획 권한 제어 강화

- 문제: 승인된 계획을 누가 수정할 수 있는지 UI와 도메인에서 엄격히 닫혀 있지 않다.
- 대상: `PlanDetail`, 수정 진입 경로, Repository/UseCase 권한 가드
- 작업:
  - 실행자 수정 가능 범위와 관리자 수정 권한 명확화
  - 비권한 사용자의 편집 진입 차단
  - 필요 시 서버 단 방어 로직 점검
- 완료 조건:
  - 승인된 계획 권한 정책이 코드에 반영된다
  - 비권한 사용자는 수정할 수 없다

## P2

### 5. 문서 및 사용자 노출 카피 정리

- 문제: README, 앱 설명, 일부 설정 문구가 아직 템플릿 또는 하드코딩 상태다.
- 대상: `README.md`, `pubspec.yaml`, 설정 화면 다국어 리소스
- 작업:
  - README를 실제 프로젝트 소개와 실행 방법 중심으로 교체
  - `pubspec` description 정리
  - 설정 화면 하드코딩 문구를 l10n으로 이관
- 완료 조건:
  - 저장소 첫 화면에서 프로젝트를 바로 이해할 수 있다
  - 사용자 노출 문구가 다국어 체계 안에 들어간다
