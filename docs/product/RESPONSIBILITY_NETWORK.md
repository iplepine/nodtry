# 책임망 관계 모델

## 결정

Nodtry의 다대다 확장은 `그룹 소셜`이 아니라 `약속별 책임망`으로 설계한다.

여러 사람이 연결될 수는 있지만, 각 약속에는 누가 당겨주는지 명확해야 한다. 책임이 "모두에게" 흩어지면 아무도 실제로 당기지 않는 문제가 생기기 때문이다.

## 핵심 가치

> 혼자 도망가지 못하게, 약속마다 붙잡아줄 사람을 붙인다.

Nodtry는 의지를 강화하는 앱이 아니다. 약속을 혼자 묻어두지 못하게 관계 구조를 바꾸는 앱이다.

## 제품 원칙

- 기본 단위는 `약속`이다.
- 각 약속에는 `실행자`와 `당기는 사람`이 있다.
- 기본값은 메인 담당자 1명이다.
- 확장은 보조 담당자 1명까지 열어 책임 공백을 줄인다.
- 사용자는 여러 사람과 연결될 수 있고, 약속마다 다른 담당자를 고를 수 있다.
- 공개 피드, 공개 랭킹, 무제한 감시자는 만들지 않는다.
- 그룹이 생기더라도 그룹은 운영 묶음이고, 실제 책임은 약속별 담당자에게 남는다.

## 관계 구조

초기:

```text
나 <-> 파트너
```

다음:

```text
나 <-> 파트너 A
나 <-> 파트너 B

공부 약속 -> 파트너 A가 당김
운동 약속 -> 파트너 B가 당김
```

이후:

```text
공부 약속 -> 메인 담당자 A + 예비 담당자 B
4주 스터디 방 -> 여러 약속을 운영상 묶는 단위
```

## 핵심 루프

1. 사용자가 작은 약속을 만든다.
2. 이 약속을 누가 당겨줄지 고른다.
3. 담당자가 승인하거나 조율한다.
4. 사용자가 오늘 실천을 처리한다.
5. 담당자가 확인, 응원, 똑똑 중 하나로 반응한다.
6. 놓친 약속은 담당자에게 남고, 필요하면 예비 담당자까지 전달된다.
7. 정산에서 누가 얼마나 당겼고, 그 뒤 행동이 이어졌는지 확인한다.

## 상태 모델 초안

### Entity

`Relation`

- `executorId`
- `managerId`
- `status`: `pending` / `active` / `rejected`

`Plan`

- `ownerId`
- `managerId`
- `status`
- `startDate`
- `endDate`

`PlanAccountabilityAssignment`

- `planId`
- `executorId`
- `primaryPullerId`
- `backupPullerIds`
- `status`: `pending` / `active` / `paused` / `ended`
- `createdAt`
- `acceptedAt`
- `endedAt`

### Event

- `responsibility_requested`
- `responsibility_accepted`
- `responsibility_declined`
- `primary_puller_changed`
- `backup_puller_added`
- `missed_promise_escalated`
- `responsibility_settled`

### Invariant

- 활성 약속은 기본적으로 메인 담당자 1명을 가진다.
- 예비 담당자는 메인 담당자의 책임을 대체하지 않는다.
- 같은 약속에 무제한 담당자를 붙이지 않는다.
- 담당자가 없는 약속은 똑똑/놓친 약속 압박을 제공하지 못한다는 점을 UI에서 알려야 한다.

## UX 계약

플랜 생성 마지막 단계의 질문은 다음 방향으로 바꾼다.

> 이 약속을 누가 당겨줄까요?

기본 선택:

- 최근 연결한 파트너
- 최근 7일 내 반응이 있었던 파트너
- 없으면 파트너 연결 CTA

담당자 표시:

- `메인 담당자`
- `예비 담당자`
- `아직 담당자 없음`

피해야 할 표현:

- 모두에게 공개
- 전체 감시
- 그룹 랭킹
- 관리 대상 목록

## 성공 지표

- 활성 플랜 중 담당자가 지정된 비율
- 파트너 피드백 후 다음 실천 전환율
- 놓친 약속이 똑똑 후 24시간 내 회수되는 비율
- 담당자 1명 대비 메인+예비 담당자 플랜의 회수율 차이
- 담당자가 부담을 느껴 확인을 중단한 비율

## 구현 순서

1. 여러 파트너 연결 제한을 제거한다.
2. 플랜 생성 시 담당 파트너 선택을 넣는다.
3. 기존 `managerId`를 선택된 담당자로 저장한다.
4. Us 탭에서 사람별로 내가 당기는 약속과 나를 당기는 약속을 분리해 보여준다.
5. 예비 담당자 모델은 파일럿 데이터 이후 추가한다.
6. 4주 스터디 방은 공개 피드가 아니라 운영 묶음으로만 설계한다.

## 이번 파일럿에서 하지 않는 것

- 공개 그룹
- 전체 공개 피드
- 랭킹
- 회사/조직 감시
- 무제한 감시자
- 자동 매칭

이번 파일럿은 여전히 10쌍으로 시작한다. 다만 학습할 질문은 `1:1이 맞는가`가 아니라, `약속이 담당자에게 묶일 때 행동이 회수되는가`로 바꾼다.
